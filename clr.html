How .NET executes your code — from source file to running program. JIT, Garbage Collection, assemblies, the type system, and more.

What is the CLR?

The Common Language Runtime (CLR) is the execution engine of the .NET platform. It is the environment that manages and runs .NET programs.

The CLR handles all the low-level services that a running program needs — so the programmer can focus on writing logic, not managing memory or platform differences.

What the CLR does for you

Memory Management
Allocates and frees memory automatically via the Garbage Collector.
JIT Compilation
Converts MSIL code to native machine code at runtime.
Type Safety
Ensures type-correct operations and prevents unsafe casts at runtime.
Exception Handling
Provides a structured, consistent mechanism for runtime errors.
Thread Management
Manages threads, synchronisation, and the thread pool.
Security
Enforces code access security and verifies assembly integrity.

Java comparison: The CLR is analogous to the JVM. MSIL/CIL is analogous to Java bytecode (.class files). JIT in CLR is analogous to JIT in the JVM.

Code Execution Flow — Step by Step

Step 1 You write C# source code in a .cs file │ ▼ Step 2 C# Compiler (csc / dotnet build) compiles it into MSIL (Microsoft Intermediate Language) stored in a .dll or .exe Assembly file │ ▼ Step 3 When you run the program, the CLR loads the Assembly │ ▼ Step 4 CLR verifies the MSIL code is type-safe and valid │ ▼ Step 5 JIT Compiler converts MSIL → native machine code (compiled method-by-method, only when first called) │ ▼ Step 6 Native code executes on the CPU CLR manages memory via Garbage Collector

MSIL / CIL — Intermediate Language

MSIL (Microsoft Intermediate Language), also called CIL (Common Intermediate Language), is a platform-independent set of instructions that all .NET languages compile to.

This is why multiple languages (C#, F#, VB.NET) can run on the same CLR — they all compile down to the same MSIL first.

Key insight: MSIL is NOT machine code. It is halfway between your source code and the CPU. The CLR's JIT compiler finishes the job at runtime by converting MSIL to actual native code for your specific machine.

C# Source → MSIL (conceptual)
// Your C# code
int a = 5;
int b = 10;
int c = a + b;

// What the compiler produces (MSIL instructions — simplified)
// ldc.i4.5        (push 5 onto stack)
// stloc.0         (store in local var a)
// ldc.i4.s 10     (push 10 onto stack)
// stloc.1         (store in local var b)
// ldloc.0         (load a)
// ldloc.1         (load b)
// add             (add them)
// stloc.2         (store in c)

JIT Compilation

JIT (Just-In-Time) Compilation is the process where the CLR converts MSIL to native machine code at runtime, just before the code is executed.

JIT compiles method by method — a method is compiled to native code the first time it is called. The compiled native code is then cached so subsequent calls are fast.

Types of JIT Compilers in .NET

TypeDescription
Normal JITDefault. Compiles methods one at a time when first called. Compiled code is cached.
Pre-JIT (NGEN / ReadyToRun)Compiles all code at install time. Faster startup but larger disk footprint.
Econo-JITOlder/constrained environments. Compiles code but may not cache it to save memory.

Modern .NET: .NET 6+ uses a technology called ReadyToRun (R2R) and tiered compilation — methods are first JIT-compiled quickly for startup speed, then re-compiled with more optimisations if they run frequently (hot paths).

Garbage Collection (GC)

The Garbage Collector (GC) is the CLR's automatic memory manager. It tracks all objects on the managed heap and frees the memory of objects that are no longer reachable by any part of the program.

You do not need to manually free memory in C# (unlike C/C++). The GC handles it automatically.

The Managed Heap and Generations

The GC organises the heap into three generations to make collection efficient:

GenerationContainsCollected
Gen 0Newly created, short-lived objectsMost frequently (cheapest)
Gen 1Objects that survived one Gen 0 collectionLess frequently
Gen 2Long-lived objects (static data, large objects)Least frequently (most expensive)
GC Cycle (simplified) Object created → stored in Gen 0 │ ▼ Gen 0 fills up → GC runs on Gen 0 │ ├─ Object unreachable → memory freed ✅ │ └─ Object still reachable → promoted to Gen 1 │ └─ If Gen 1 fills → promoted to Gen 2

IDisposable and Deterministic Cleanup

The GC runs non-deterministically — you can't predict exactly when it runs. For resources that need to be released immediately (file handles, DB connections), implement IDisposable and use the using statement.

IDisposable Pattern
class ResourceHolder : IDisposable
{
    private bool _disposed = false;

    public void Dispose()
    {
        if (!_disposed)
        {
            // Release unmanaged resources here
            _disposed = true;
        }
    }
}

// Dispose() called automatically at end of using block
using (ResourceHolder rh = new ResourceHolder())
{
    // use rh
}  // rh.Dispose() called here automatically

CTS — Common Type System

The Common Type System (CTS) defines a standard set of data types and rules for how types are declared, used, and managed in the .NET runtime.

CTS ensures that types from different .NET languages are compatible — an int in C# is the same underlying type as an Integer in VB.NET.

CTS Type Hierarchy

System.Object ← everything inherits from this │ ├── Value Types (stored on stack, copied by value) │ ├── Structs (int, double, bool, char, DateTime...) │ └── Enumerations (enum keyword) │ └── Reference Types (stored on heap, hold a reference/pointer) ├── Classes (class keyword) ├── Interfaces (interface keyword) ├── Arrays └── Delegates

CLS — Common Language Specification

The Common Language Specification (CLS) is a set of rules and guidelines that a .NET language or library must follow to be fully interoperable with all other .NET languages.

CLS is a subset of CTS — it defines the minimum feature set that all .NET languages must support.

Simple analogy: CTS is the full vocabulary. CLS is the common vocabulary that everyone agrees to use when talking to each other. If your library follows CLS rules, any .NET language (C#, VB.NET, F#) can use it.

CTSCLS
What it isFull .NET type systemRules for cross-language compatibility
ScopeAll .NET types and featuresMinimum shared subset
Who follows itThe CLR itselfPublic APIs meant to be cross-language
Example rule—No unsigned types (uint) in public APIs — not all languages support them

Assemblies

An Assembly is the fundamental unit of deployment and versioning in .NET. When you compile a .NET project, the output is an assembly — a .dll or .exe file.

Namespaces

A Namespace is a logical grouping of related classes, interfaces, and types. It prevents naming conflicts and organises code.

C#
// Declaring a namespace
namespace MyApp.Models
{
    class Student
    {
        public string Name { get; set; }
    }
}

// Using it elsewhere
using MyApp.Models;           // import the namespace

Student s = new Student();   // no need for full name
// OR without using:
MyApp.Models.Student s2 = new MyApp.Models.Student();

Value Types vs Reference Types

One of the most important distinctions in C# — understanding this prevents many confusing bugs.

FeatureValue TypesReference Types
Stored inStackHeap (reference on stack)
Assignment copiesThe actual valueThe reference (pointer)
Default value0, false, '\0' etc.null
Examplesint, double, bool, struct, enumclass, string, array, interface
Value Type vs Reference Type Behaviour
// Value type — copy is independent
int a = 10;
int b = a;      // b gets a COPY of 10
b = 20;
Console.WriteLine(a);   // 10 — unchanged
Console.WriteLine(b);   // 20

// Reference type — both variables point to the same object
int[] arr1 = { 1, 2, 3 };
int[] arr2 = arr1;      // arr2 points to the SAME array
arr2[0] = 99;
Console.WriteLine(arr1[0]);  // 99 — arr1 is also changed!
