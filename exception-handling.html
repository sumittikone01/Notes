Exception
Handling

Everything about handling runtime errors in C# — from the class hierarchy and try-catch mechanics to custom exceptions and resource management.

What is an Exception?

An Exception is an unwanted or unexpected event that occurs during program execution and disrupts the normal flow of the program.

Exceptions in C# usually occur due to problems in program logic or invalid operations at runtime. Unlike system-level errors (like StackOverflowException), most exceptions are within the programmer's control and can be handled.

The process of responding to exceptions is called Exception Handling.

Key difference from Java: C# has no checked exceptions. The compiler never forces you to handle or declare any exception. All exceptions in C# behave like Java's unchecked (runtime) exceptions.

Exception Class Hierarchy

Every exception in C# ultimately inherits from System.Exception.

System.Object └── System.Exception ├── System.SystemException │ ├── System.ArithmeticException │ │ └── System.DivideByZeroException │ ├── System.NullReferenceException │ ├── System.IndexOutOfRangeException │ ├── System.InvalidCastException │ ├── System.StackOverflowException │ ├── System.OutOfMemoryException │ ├── System.IO.IOException │ │ └── System.IO.FileNotFoundException │ └── System.FormatException │ └── System.ApplicationException └── YourCustomException ← user-defined

Points to remember: System.Object is the root of everything in C#. System.SystemException is the base for all built-in CLR exceptions. System.ApplicationException is the conventional base for user-defined app-level exceptions.

Types of Exceptions in C#

C# does not distinguish between checked and unchecked exceptions like Java. All exceptions are unchecked. They are logically grouped as:

Created by the programmer to represent application-specific errors. Should inherit from Exception or ApplicationException. Covered fully in the Custom Exceptions section.

try and catch Block

try block — contains risky code that may throw an exception.
catch block — handles the exception. Executes only when an exception is thrown.

Syntax — C#
try
{
    // Risky code that may throw an exception
}
catch (ExceptionType referenceVariable)
{
    // Code to handle the exception
}
Example — Division by Zero
using System;

class Program
{
    static void Main(string[] args)
    {
        Console.WriteLine("Program Starts");

        Console.Write("Enter first number: ");
        int a = int.Parse(Console.ReadLine());

        Console.Write("Enter second number: ");
        int b = int.Parse(Console.ReadLine());

        try
        {
            int result = a / b;
            Console.WriteLine("Result: " + result);
        }
        catch (DivideByZeroException e)
        {
            Console.WriteLine("Cannot divide by zero. " + e.Message);
        }

        Console.WriteLine("Program Ends");
    }
}

Key behaviour: After the exception is caught and handled, execution continues from after the catch block — the program does not crash. Lines after the try-catch block execute normally.

Useful Properties of the Exception Class

PropertyDescription
e.MessageShort description of the exception
e.StackTraceCall stack at the point the exception was thrown
e.SourceName of the application or object that caused the error
e.InnerExceptionThe original exception that caused the current one (if any)
e.GetType().NameRuntime type name of the exception
Example
try
{
    int[] arr = new int[3];
    arr[10] = 5;   // IndexOutOfRangeException
}
catch (IndexOutOfRangeException e)
{
    Console.WriteLine("Message     : " + e.Message);
    Console.WriteLine("Source      : " + e.Source);
    Console.WriteLine("Stack Trace : " + e.StackTrace);
}

Multiple catch Blocks

A single try block can be followed by multiple catch blocks, each handling a different type of exception.

Syntax
try
{
    // Risky code
}
catch (ExceptionType1 e1)
{
    // Handle ExceptionType1
}
catch (ExceptionType2 e2)
{
    // Handle ExceptionType2
}
catch (Exception e)
{
    // Generic fallback — handles anything not caught above
}
Example
using System;

class MultiCatchDemo
{
    static void Main(string[] args)
    {
        string str = "Akshay";
        int[] arr = { 20, 60, 80, 90, 50 };

        Console.Write("Enter an index: ");
        int n = int.Parse(Console.ReadLine());

        try
        {
            Console.WriteLine(arr[n]);       // may throw IndexOutOfRangeException
            Console.WriteLine(str[n]);       // may throw IndexOutOfRangeException
            Console.WriteLine(100 / n);      // may throw DivideByZeroException
        }
        catch (DivideByZeroException e)
        {
            Console.WriteLine("Cannot divide by zero: " + e.Message);
        }
        catch (IndexOutOfRangeException e)
        {
            Console.WriteLine("Index is out of range: " + e.Message);
        }
        catch (Exception e)
        {
            Console.WriteLine("Something went wrong: " + e.Message);
        }
    }
}

Important Rules: Catch blocks are checked top to bottom — first match executes. Always place specific (child) exceptions before general (parent) ones. Placing catch (Exception e) first makes all specific blocks below it unreachable — C# flags this as a compile-time error.

finally Block

The finally block always executes after the try block — whether an exception occurred or not.

Uses: Release resources (files, DB connections, streams) · Cleanup tasks · Guarantee execution of critical code.

Syntax
try   { /* risky code */        }
catch (ExceptionType e) { /* handle */          }
finally               { /* always runs */     }
Example
try
{
    Console.WriteLine("Inside try block");
    int result = 10 / 2;    // change to 10/0 to trigger exception
    Console.WriteLine("Result: " + result);
}
catch (DivideByZeroException e)
{
    Console.WriteLine("Exception caught: " + e.Message);
}
finally
{
    Console.WriteLine("Finally block always executes — cleanup done.");
}

Note: finally executes even if a return statement is inside try or catch. It does NOT execute only if Environment.Exit() is called, or a fatal crash (like StackOverflowException) kills the process.

Exception Filters — when Clause (C# 6+)

The when keyword adds a condition to a catch block. The catch only executes if the condition is true. This is a C#-only feature not present in Java.

Syntax
catch (ExceptionType e) when (condition)
{
    // executes only if condition is true
}
Example — HTTP error codes
int[] codes = { 404, 500, 200 };

foreach (int code in codes)
{
    try
    {
        if (code != 200)
            throw new Exception("HTTP Error: " + code);
        Console.WriteLine("Success: " + code);
    }
    catch (Exception e) when (e.Message.Contains("404"))
    {
        Console.WriteLine("Not Found: " + e.Message);
    }
    catch (Exception e) when (e.Message.Contains("500"))
    {
        Console.WriteLine("Server Error: " + e.Message);
    }
}

You can also use when to catch multiple unrelated exception types in one block:

Multi-type catch using when
catch (Exception e) when (e is FormatException || e is OverflowException)
{
    Console.WriteLine("Invalid input: " + e.Message);
}

Advantage of when: The filter runs before the stack is unwound, which means you keep the full original stack trace for debugging — you don't lose context like you might with a re-throw.

throw Keyword

The throw keyword is used to explicitly throw an exception — either a brand new one or an existing caught one.

Throwing a new exception
static void CheckAge(int age)
{
    if (age < 18)
        throw new ArgumentException("Age must be 18 or above. Got: " + age);

    Console.WriteLine("Access granted.");
}

static void Main(string[] args)
{
    try   { CheckAge(15); }
    catch (ArgumentException e) { Console.WriteLine("Caught: " + e.Message); }
}

Re-throwing an Exception

Bare throw vs throw e
catch (Exception e)
{
    Console.WriteLine("Logging: " + e.Message);
    throw;        // ✅ Preserves original stack trace — use this
}

catch (Exception e)
{
    throw e;      // ❌ Resets stack trace — avoid this
}

catch (IOException e)
{
    // Wrap in a higher-level exception (original becomes InnerException)
    throw new InvalidOperationException("Higher-level message", e);
}

Remember: Always use bare throw; when re-throwing — it preserves the original stack trace. throw e; resets it and makes debugging much harder.

using Statement — Auto Resource Management

The using statement ensures objects implementing IDisposable are automatically closed/disposed when the block ends — even if an exception occurs.

This is C#'s equivalent of Java's try-with-resources.

Syntax
using (ResourceType resource = new ResourceType())
{
    // use the resource
}  // resource.Dispose() is called automatically here
Example — Reading a File
using System;
using System.IO;

class UsingDemo
{
    static void Main(string[] args)
    {
        Console.WriteLine("----- App Started -----");

        try
        {
            using (StreamReader sr = new StreamReader("test.txt"))
            {
                string line;
                while ((line = sr.ReadLine()) != null)
                    Console.WriteLine(line);
            }  // sr.Dispose() called here automatically
        }
        catch (IOException e)
        {
            Console.WriteLine("IOException: " + e.Message);
        }

        Console.WriteLine("----- App Finished -----");
    }
}

C# 8+ — using Declaration (Simplified)

C# 8+ Syntax — no braces needed
static void Main(string[] args)
{
    using StreamReader sr = new StreamReader("test.txt");
    // sr is disposed when it goes out of scope (end of method)
    string content = sr.ReadToEnd();
    Console.WriteLine(content);
}

Any class implementing IDisposable can be used with using: StreamReader, SqlConnection, HttpClient, FileStream, and your own classes if they implement the interface.

User-Defined Custom Exceptions

When built-in exceptions don't describe your specific error condition, create a custom exception class.

Steps to Create a Custom Exception

  1. Create a class that extends Exception (or ApplicationException).
  2. Define constructors — at minimum a default one and one accepting a message string.
  3. Use the throw keyword to throw it when the specific condition occurs.
  4. Handle it using try-catch in the calling code.
Example — Banking Application
using System;

// Step 1: Define the custom exception
class InsufficientBalanceException : Exception
{
    public InsufficientBalanceException() : base("Insufficient balance.") { }

    public InsufficientBalanceException(string message) : base(message) { }

    public InsufficientBalanceException(string message, Exception inner)
        : base(message, inner) { }
}

// Step 2: Use it in a class
class BankAccount
{
    private double balance;

    public BankAccount(double balance) { this.balance = balance; }

    public void Withdraw(double amount)
    {
        if (amount > balance)
            throw new InsufficientBalanceException(
                "Withdrawal failed! ₹" + amount + " exceeds balance ₹" + balance);

        balance -= amount;
        Console.WriteLine("Successful. Remaining: ₹" + balance);
    }
}

// Step 3: Handle the custom exception
class Program
{
    static void Main(string[] args)
    {
        BankAccount account = new BankAccount(5000);

        try { account.Withdraw(6000); }   // exceeds balance
        catch (InsufficientBalanceException e)
            { Console.WriteLine("Caught: " + e.Message); }

        try { account.Withdraw(3000); }   // valid
        catch (InsufficientBalanceException e)
            { Console.WriteLine("Caught: " + e.Message); }
    }
}

Recommended Custom Exception Template

Best-practice template
[Serializable]
public class MyCustomException : Exception
{
    public MyCustomException() { }

    public MyCustomException(string message)
        : base(message) { }

    public MyCustomException(string message, Exception innerException)
        : base(message, innerException) { }
}

The [Serializable] attribute and three-constructor pattern is the recommended .NET standard for custom exceptions.

Exception Propagation

C# has no throws keyword. If a method does not handle an exception, it propagates automatically up the call stack to the caller.

Example
static void MethodC() { throw new InvalidOperationException("Broke in C!"); }
static void MethodB() { MethodC(); }   // propagates up
static void MethodA() { MethodB(); }   // propagates up

static void Main(string[] args)
{
    try   { MethodA(); }
    catch (InvalidOperationException e)
        { Console.WriteLine("Caught in Main: " + e.Message); }
}
MethodC() ← exception thrown here ↑ MethodB() ← propagates up ↑ MethodA() ← propagates up ↑ Main() ← caught here ✅

If the exception is not caught anywhere in the call stack, the CLR terminates the program with an unhandled exception error.

InnerException — Wrapping Exceptions

When catching an exception and throwing a new one, you can wrap the original as an InnerException. This preserves the root cause for debugging.

Example
static void LoadConfig()
{
    try
    {
        string data = File.ReadAllText("config.json");
    }
    catch (IOException e)
    {
        throw new ApplicationException("Failed to load config.", e);
        // original IOException is now e.InnerException
    }
}

static void Main(string[] args)
{
    try   { LoadConfig(); }
    catch (ApplicationException e)
    {
        Console.WriteLine("Error     : " + e.Message);
        Console.WriteLine("Root Cause: " + e.InnerException?.Message);
    }
}
