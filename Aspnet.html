ASP.NET Core Web API

A complete reference for building Web APIs with ASP.NET Core — from HTTP basics and controllers to middleware, dependency injection, EF Core, and authentication.

What is ASP.NET Core Web API?

ASP.NET Core is Microsoft's modern, cross-platform web framework built on .NET. A Web API (Application Programming Interface) is a service that listens for HTTP requests and returns data (usually JSON) instead of HTML pages.

Web APIs are used as the backend for mobile apps, front-end web apps (React, Angular), other services, and any system that needs to exchange data over a network.

REST — The Standard Architecture

Most ASP.NET Core Web APIs follow the REST (Representational State Transfer) style. REST uses standard HTTP methods to perform operations on resources (data entities like users, products, orders).

Example: A Product resource at /api/products can be fetched with GET, created with POST, updated with PUT, and deleted with DELETE — all using the same URL, just different HTTP methods.

HTTP Basics — Request & Response

Every API interaction is an HTTP Request sent by the client and an HTTP Response returned by the server.

Client (Browser / Mobile App / Postman) │ │ HTTP Request │ ┌─────────────────────────────────────────────────┐ │ │ Method: GET │ │ │ URL: https://api.example.com/api/products/3 │ │ │ Headers: Accept: application/json │ │ │ Body: (empty for GET) │ │ └─────────────────────────────────────────────────┘ ▼ ASP.NET Core Web API Server │ │ HTTP Response │ ┌─────────────────────────────────────────────────┐ │ │ Status: 200 OK │ │ │ Headers: Content-Type: application/json │ │ │ Body: {"id":3,"name":"Laptop","price":999.99} │ │ └─────────────────────────────────────────────────┘ ▼ Client receives and uses the JSON data

HTTP Methods (Verbs)

Each HTTP method tells the server what action to perform on the resource.

MethodCRUD ActionDescriptionHas Body?
GETReadRetrieve a resource or list of resourcesNo
POSTCreateCreate a new resource with data in the bodyYes
PUTUpdate (full)Replace a resource completelyYes
PATCHUpdate (partial)Update only specific fields of a resourceYes
DELETEDeleteRemove a resourceNo

Idempotent means calling the same request multiple times gives the same result. GET, PUT, DELETE are idempotent. POST is not — each call creates a new resource.

Program.cs — App Entry Point

Program.cs is where your application starts. It does two things: registers services (DI container) and configures the middleware pipeline.

Program.cs — ASP.NET Core 6+
var builder = WebApplication.CreateBuilder(args);

// ── 1. Register Services (Dependency Injection) ──
builder.Services.AddControllers();                // adds MVC controllers
builder.Services.AddEndpointsApiExplorer();        // for Swagger
builder.Services.AddSwaggerGen();                  // Swagger docs
builder.Services.AddDbContext<AppDbContext>(opts =>
    opts.UseSqlServer(builder.Configuration.GetConnectionString("Default")));
builder.Services.AddScoped<IProductService, ProductService>();

var app = builder.Build();

// ── 2. Configure Middleware Pipeline ──
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();
app.UseAuthentication();   // who are you?
app.UseAuthorization();     // are you allowed?
app.MapControllers();       // map routes to controller actions

app.Run();

Two sections, two phases: Everything before builder.Build() is service registration. Everything after is pipeline configuration. Order matters in the pipeline — middleware runs top to bottom on every request.

Controllers & Actions

A Controller is a class that handles incoming HTTP requests and returns responses. Each public method inside a controller that handles a request is called an Action.

ProductsController.cs — Full CRUD
using Microsoft.AspNetCore.Mvc;

[ApiController]                       // enables auto model validation & binding
[Route("api/[controller]")]           // → route: /api/products
public class ProductsController : ControllerBase
{
    private readonly IProductService _service;

    // Service is injected via constructor (Dependency Injection)
    public ProductsController(IProductService service)
        => _service = service;

    // GET /api/products
    [HttpGet]
    public IActionResult GetAll()
    {
        var products = _service.GetAll();
        return Ok(products);             // 200 OK + JSON body
    }

    // GET /api/products/5
    [HttpGet("{id}")]
    public IActionResult GetById(int id)
    {
        var product = _service.GetById(id);
        if (product == null) return NotFound();   // 404
        return Ok(product);
    }

    // POST /api/products
    [HttpPost]
    public IActionResult Create([FromBody] CreateProductDto dto)
    {
        var created = _service.Create(dto);
        return CreatedAtAction(nameof(GetById),   // 201 Created
            new { id = created.Id }, created);
    }

    // PUT /api/products/5
    [HttpPut("{id}")]
    public IActionResult Update(int id, [FromBody] UpdateProductDto dto)
    {
        var success = _service.Update(id, dto);
        if (!success) return NotFound();
        return NoContent();   // 204 No Content
    }

    // DELETE /api/products/5
    [HttpDelete("{id}")]
    public IActionResult Delete(int id)
    {
        var success = _service.Delete(id);
        if (!success) return NotFound();
        return NoContent();   // 204 No Content
    }
}

ControllerBase vs Controller: Use ControllerBase for Web API (no View support needed). Use Controller (inherits ControllerBase) only when returning MVC Views like View().

Routing

Routing maps an incoming HTTP request URL to a specific controller action. ASP.NET Core supports Attribute Routing (used in Web APIs).

Route Tokens

TokenReplaced byExample
[controller]Controller class name (minus "Controller")ProductsController → products
[action]Method nameGetAll → getall
{id}Route parameter/api/products/5 → id=5
{id?}Optional route parameter/api/products or /api/products/5
{id:int}Constrained parameter (integers only)/api/products/abc → 404
Routing Examples
[Route("api/[controller]")]         // /api/products
[Route("api/v1/products")]          // explicit route
[Route("api/[controller]/[action]")] // /api/products/getall

[HttpGet("{id:int}")]               // only matches integers
[HttpGet("{name:alpha}")]           // only matches letters
[HttpGet("{id:int:min(1)}")]        // integer, must be ≥ 1

// Multiple routes on one action
[HttpGet("{id}")]
[HttpGet("details/{id}")]
public IActionResult GetById(int id) { ... }

Query Strings

Query string parameters
// GET /api/products?page=2&pageSize=10&search=laptop
[HttpGet]
public IActionResult GetAll(
    [FromQuery] int page = 1,
    [FromQuery] int pageSize = 10,
    [FromQuery] string? search = null)
{
    var results = _service.GetPaged(page, pageSize, search);
    return Ok(results);
}

IActionResult — Returning Responses

IActionResult is the return type for controller actions. It lets you return any HTTP response — with a status code and optional body.

Helper MethodStatus CodeWhen to use
Ok(data)200Successful GET, PUT — return data
Created(url, data)201Resource created via POST
CreatedAtAction(...)201POST — return 201 + Location header pointing to GET
NoContent()204PUT / DELETE success, no data to return
BadRequest(obj)400Validation failed, bad input
Unauthorized()401Not logged in
Forbid()403Logged in but no permission
NotFound()404Resource doesn't exist
Conflict()409Resource already exists
StatusCode(code, data)AnyCustom status code

ActionResult<T> — Typed Return (Preferred)

Strongly-typed action result
// ActionResult<T> gives you both type safety and HTTP flexibility
[HttpGet("{id}")]
public ActionResult<ProductDto> GetById(int id)
{
    var product = _service.GetById(id);

    if (product == null)
        return NotFound(
            new { Message = $"Product {id} not found" });

    return product;      // implicit 200 OK + serialized ProductDto
}

Models & DTOs

Model (Entity)

A Model represents your database entity — it maps directly to a database table (when using EF Core).

Models/Product.cs
public class Product
{
    public int     Id          { get; set; }
    public string  Name        { get; set; } = string.Empty;
    public decimal Price       { get; set; }
    public int     Stock       { get; set; }
    public bool    IsAvailable { get; set; }
    public DateTime CreatedAt  { get; set; }
}

DTO (Data Transfer Object)

A DTO is a trimmed version of a model, used to control exactly what data enters and exits your API. Never expose your raw entity directly — use DTOs.

DTOs/ProductDto.cs
// What the API accepts when CREATING a product
public class CreateProductDto
{
    [Required]
    [MaxLength(100)]
    public string  Name  { get; set; } = string.Empty;

    [Range(0.01, 99999.99)]
    public decimal Price { get; set; }

    [Range(0, int.MaxValue)]
    public int     Stock { get; set; }
}

// What the API returns when READING a product
public class ProductResponseDto
{
    public int     Id    { get; set; }
    public string  Name  { get; set; } = string.Empty;
    public decimal Price { get; set; }
    // Notice: no Stock, no internal fields exposed
}

Why DTOs? They protect against over-posting attacks, hide internal fields, allow the API contract and database schema to evolve independently, and simplify validation.

Data Annotations & Validation

Data annotations are attributes placed on DTO properties to declare validation rules. With [ApiController], validation runs automatically and returns 400 Bad Request if any rule fails.

AnnotationDescription
[Required]Field must be present and non-null
[MaxLength(n)]String length ≤ n
[MinLength(n)]String length ≥ n
[StringLength(max, MinimumLength=min)]String length between min and max
[Range(min, max)]Numeric value within range
[EmailAddress]Valid email format
[Url]Valid URL format
[RegularExpression("pattern")]Must match regex pattern
[Compare("OtherProp")]Must equal value of another property (e.g., confirm password)
Validation example
public class RegisterDto
{
    [Required(ErrorMessage = "Name is required")]
    [StringLength(50, MinimumLength = 2)]
    public string Name { get; set; } = "";

    [Required]
    [EmailAddress(ErrorMessage = "Invalid email format")]
    public string Email { get; set; } = "";

    [Required]
    [MinLength(8, ErrorMessage = "Password min 8 characters")]
    public string Password { get; set; } = "";

    [Compare("Password", ErrorMessage = "Passwords do not match")]
    public string ConfirmPassword { get; set; } = "";
}

// Controller action — [ApiController] handles validation automatically
[HttpPost("register")]
public IActionResult Register([FromBody] RegisterDto dto)
{
    // If any annotation fails → automatic 400 BadRequest with error details
    // If you're NOT using [ApiController], check manually:

    ...

Middleware Pipeline

Middleware is software assembled into the request pipeline that processes every HTTP request and response. Each middleware component either handles the request or passes it to the next component in the chain.

Incoming HTTP Request │ ▼ ┌──────────────────────────────┐ │ UseExceptionHandler │ ← catches unhandled exceptions └──────────────────┬───────────┘ ▼ ┌──────────────────────────────┐ │ UseHttpsRedirection │ ← http → https └──────────────────┬───────────┘ ▼ ┌──────────────────────────────┐ │ UseStaticFiles │ ← serves wwwroot files └──────────────────┬───────────┘ ▼ ┌──────────────────────────────┐ │ UseCors │ ← Cross-Origin access └──────────────────┬───────────┘ ▼ ┌──────────────────────────────┐ │ UseAuthentication │ ← who are you? └──────────────────┬───────────┘ ▼ ┌──────────────────────────────┐ │ UseAuthorization │ ← are you allowed? └──────────────────┬───────────┘ ▼ ┌──────────────────────────────┐ │ MapControllers │ ← routes request to controller action └──────────────────┬───────────┘ ▼ Controller Action executes │ Response travels back UP through the same pipeline

Custom Middleware

Custom logging middleware
public class RequestLoggingMiddleware
{
    private readonly RequestDelegate _next;

    public RequestLoggingMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext context)
    {
        Console.WriteLine($"→ {context.Request.Method} {context.Request.Path}");

        await _next(context);    // pass to next middleware

        Console.WriteLine($"← {context.Response.StatusCode}");
    }
}

// Register in Program.cs:
app.UseMiddleware<RequestLoggingMiddleware>();

Dependency Injection (DI)

Dependency Injection is a design pattern where a class receives its dependencies from the outside (injected by the framework) rather than creating them itself. ASP.NET Core has DI built in.

DI in action — 3 simple steps
// Step 1: Define interface
public interface IEmailService
{
    void SendEmail(string to, string subject, string body);
}

// Step 2: Implement it
public class SmtpEmailService : IEmailService
{
    public void SendEmail(string to, string subject, string body)
        => Console.WriteLine($"Sending email to {to}: {subject}");
}

// Step 3: Register in Program.cs
builder.Services.AddScoped<IEmailService, SmtpEmailService>();

// Now inject anywhere via constructor
public class UserController : ControllerBase
{
    private readonly IEmailService _email;

    public UserController(IEmailService email) => _email = email;

    [HttpPost("register")]
    public IActionResult Register([FromBody] RegisterDto dto)
    {
        _email.SendEmail(dto.Email, "Welcome!", "Thanks for registering.");
        return Ok();
    }
}

Service Lifetimes

When registering services, you choose a lifetime that controls how long each instance lives.

LifetimeRegistrationInstance CreatedUse When
TransientAddTransient<I, C>()Every time it's requested (new instance per injection)Lightweight, stateless services
ScopedAddScoped<I, C>()Once per HTTP request (shared within same request)Database context, unit of work
SingletonAddSingleton<I, C>()Once for the entire app lifetime (shared globally)Caches, configuration, shared state

⚠ Captive dependency: Never inject a Scoped or Transient service into a Singleton — the short-lived service becomes "captured" and lives as long as the Singleton, which causes bugs. EF Core's DbContext is Scoped — never inject it into a Singleton.

Entity Framework Core (EF Core)

EF Core is Microsoft's Object-Relational Mapper (ORM). It lets you work with a database using C# objects — no raw SQL needed (unless you want it).

DbContext — The Database Bridge

Data/AppDbContext.cs
using Microsoft.EntityFrameworkCore;

public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options) { }

    // Each DbSet maps to a table
    public DbSet<Product>  Products  { get; set; }
    public DbSet<Category> Categories { get; set; }
    public DbSet<Order>    Orders    { get; set; }
}

CRUD with EF Core

Full CRUD operations
public class ProductService : IProductService
{
    private readonly AppDbContext _db;
    public ProductService(AppDbContext db) => _db = db;

    // READ all
    public async Task<List<Product>> GetAllAsync()
        => await _db.Products.ToListAsync();

    // READ one
    public async Task<Product?> GetByIdAsync(int id)
        => await _db.Products.FindAsync(id);

    // CREATE
    public async Task<Product> CreateAsync(CreateProductDto dto)
    {
        var product = new Product { Name = dto.Name, Price = dto.Price };
        _db.Products.Add(product);
        await _db.SaveChangesAsync();
        return product;
    }

    // UPDATE
    public async Task<bool> UpdateAsync(int id, UpdateProductDto dto)
    {
        var product = await _db.Products.FindAsync(id);
        if (product == null) return false;
        product.Name  = dto.Name;
        product.Price = dto.Price;
        await _db.SaveChangesAsync();
        return true;
    }

    // DELETE
    public async Task<bool> DeleteAsync(int id)
    {
        var product = await _db.Products.FindAsync(id);
        if (product == null) return false;
        _db.Products.Remove(product);
        await _db.SaveChangesAsync();
        return true;
    }
}

Swagger / OpenAPI

Swagger (OpenAPI) automatically generates interactive API documentation from your controllers. It provides a browser UI where you can see all your endpoints and test them — no Postman needed.

Program.cs — Swagger setup
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen(c =>
{
    c.SwaggerDoc("v1", new OpenApiInfo
    {
        Title   = "My API",
        Version = "v1",
        Description = "ASP.NET Core Web API"
    });

    // Include XML comments for endpoint descriptions
    var xmlFile = $"{Assembly.GetExecutingAssembly().GetName().Name}.xml";
    var xmlPath = Path.Combine(AppContext.BaseDirectory, xmlFile);
    c.IncludeXmlComments(xmlPath);
});

// In pipeline:
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI(c => c.SwaggerEndpoint("/swagger/v1/swagger.json", "v1"));
}
// Visit: https://localhost:PORT/swagger
XML doc comments for Swagger
/// <summary>Get a product by its ID</summary>
/// <param name="id">The product ID</param>
/// <response code="200">Product found</response>
/// <response code="404">Product not found</response>
[HttpGet("{id}")]
[ProducesResponseType(typeof(ProductResponseDto), 200)]

CORS — Cross-Origin Resource Sharing

CORS is a browser security mechanism that blocks a web page from making requests to a different domain than the one that served it. You need to configure CORS in your API to allow your front-end (e.g., React on localhost:3000) to call your API.

CORS Configuration in Program.cs
// Register CORS policy
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowFrontend", policy =>
    {
        policy.WithOrigins("http://localhost:3000", "https://myapp.com")
              .AllowAnyHeader()
              .AllowAnyMethod()
              .AllowCredentials();
    });

    // For development — allow everything (NOT for production)
    options.AddPolicy("AllowAll", policy =>
        policy.AllowAnyOrigin().AllowAnyHeader().AllowAnyMethod());
});

// Apply in pipeline (must come before UseAuthorization)
app.UseCors("AllowFrontend");

Action Filters

Action Filters let you run code before or after a controller action executes — without touching the action itself. Perfect for cross-cutting concerns like logging, validation, and timing.

Filter TypeRuns When
IActionFilterBefore & after action method executes
IResultFilterBefore & after result (IActionResult) executes
IExceptionFilterWhen an unhandled exception occurs
IAuthorizationFilterBefore anything else — authorization check
Custom exception filter
public class GlobalExceptionFilter : IExceptionFilter
{
    public void OnException(ExceptionContext context)
    {
        context.Result = new ObjectResult(new
        {
            Error   = context.Exception.Message,
            Type    = context.Exception.GetType().Name,
            Status  = 500
        }) { StatusCode = 500 };

        context.ExceptionHandled = true;
    }
}

// Register globally in Program.cs
builder.Services.AddControllers(options =>
    options.Filters.Add<GlobalExceptionFilter>());

// Or use on specific controller/action
[ServiceFilter(typeof(GlobalExceptionFilter))]
public class ProductsController : ControllerBase { ... }

Authentication & Authorisation Basics

ASP.NET Core separates Authentication (who are you?) from Authorisation (what are you allowed to do?). JWT (JSON Web Token) is the most common approach for Web APIs.

JWT setup in Program.cs
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer           = true,
            ValidateAudience         = true,
            ValidateLifetime         = true,
            ValidateIssuerSigningKey = true,
            ValidIssuer              = builder.Configuration["Jwt:Issuer"],
            ValidAudience            = builder.Configuration["Jwt:Audience"],
            IssuerSigningKey         = new SymmetricSecurityKey(
                Encoding.UTF8.GetBytes(builder.Configuration["Jwt:Key"]!))
        };
    });
Protecting endpoints
[ApiController]
[Route("api/[controller]")]
[Authorize]                          // ← ALL actions require auth
public class OrdersController : ControllerBase
{
    [HttpGet]
    public IActionResult GetMyOrders() { ... }

    [HttpGet("admin/all")]
    [Authorize(Roles = "Admin")]       // ← only Admin role
    public IActionResult GetAllOrders() { ... }

    [HttpGet("public-info")]
    [AllowAnonymous]                   // ← override — no auth needed
    public IActionResult PublicInfo() { ... }
}
