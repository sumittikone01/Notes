## 1. Why Do We Need to Pass Data?

* In MVC, the **Controller** fetches or prepares data (from a database, calculation, etc.) and the **View** displays it as HTML.
* But the View cannot go get data on its own. The Controller must **hand the data over** to the View.

```
┌─────────────────────────────────────────────────────────┐
│                    CONTROLLER                           │
│                                                         │
│  var name = "Akash";   ← Data is here                   │
│  var price = 15000;                                     │
│                                                         │
│  // How do we give this to the View?                    │
│  return View();   ← View gets nothing without a method  │
└─────────────────────────────────────────────────────────┘
                          │
            We need a way to hand data
            from Controller → View
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                      VIEW (.cshtml)                     │
│  Needs the data to display it as HTML                   │
└─────────────────────────────────────────────────────────┘
```

ASP.NET MVC gives us **4 ways** to do this:

```
Controller ──────────────────────────────► View
              │
              ├── 1. Model       (best way — strongly typed)
              ├── 2. ViewData    (dictionary — key/value)
              ├── 3. ViewBag     (dynamic — dot notation)
              └── 4. TempData    (survives redirect)
```

## 2. The 4 Methods — Overview

```
┌────────────────┬──────────────────────────────┬─────────────────────────┬──────────────────────┐
│  Method        │  What it is                  │  Typed?                 │  Survives Redirect?  │
├────────────────┼──────────────────────────────┼─────────────────────────┼──────────────────────┤
│  Model         │  Pass a C# object directly   │  ✅ Strongly typed      │  ❌ No               │
│  ViewData      │  Dictionary ["key"] = value  │  ❌ Loosely typed       │  ❌ No               │
│  ViewBag       │  Dynamic property .Property  │  ❌ Loosely typed       │  ❌ No               │
│  TempData      │  Dictionary — session backed │  ❌ Loosely typed       │  ✅ YES              │
└────────────────┴──────────────────────────────┴─────────────────────────┴──────────────────────┘
```

**"Strongly Typed"** means the compiler checks your code at build time. If you misspell a property, it shows an error immediately.

**"Loosely Typed"** means no compile-time check. If you misspell a key, it silently returns `null` at runtime — harder to catch bugs.

## 3. Method 1 — Model (Strongly Typed)

> **This is the recommended and best way** to pass data.
>
> You pass a C# object directly to the View, and the View declares what type it expects at the top using `@model`.

### How it Works

```
Controller                            View
────────────────                      ─────────────────────────
var p = new Product()                 @model Product
{                                 
    prodId   = 1,                     @Model.prodId
    prodName = "Intel i5 CPU",        @Model.prodName
    prodPrice= 15000,                 @Model.prodPrice
    prodQty  = 20                     @Model.prodQty
};
return View(p);  ◄── pass object
```

### Step 1 — Create the Model

```csharp
// Models/Product.cs
namespace MyMvcApp.Models
{
    public class Product
    {
        public int    prodId    { get; set; }
        public string prodName  { get; set; } = "";
        public int    prodPrice { get; set; }
        public int    prodQty   { get; set; }
    }
}
```

### Step 2 — Controller sets data and passes to View

```csharp
// Controllers/ProductController.cs
public IActionResult Index()
{
    // Create the object and fill it with data
    Product p = new Product
    {
        prodId    = 1,
        prodName  = "Intel i5 CPU",
        prodPrice = 15000,
        prodQty   = 20
    };

    return View(p);   // ← Pass the object to the View
}
```

### Step 3 — View receives and displays it

```html
<!-- Views/Product/Index.cshtml -->
@model MyMvcApp.Models.Product    ← Declares: "I expect a Product object"

<h1>Product Details</h1>
<hr />
<div class="text-center">
    <font size="18px">
        Product Id:    @Model.prodId   <br />
        Product Name:  @Model.prodName <br />
        Product Price: @Model.prodPrice<br />
        Product Qty:   @Model.prodQty  <br />
    </font>
</div>
```

### Passing a List (Multiple Records)

When you have multiple items (e.g., list of students from DB), use `IEnumerable<Model>`:

```csharp
// Controller — pass a list
public IActionResult Index()
{
    List<Student> students = new List<Student>
    {
        new Student { roll = 1, name = "Akash",  gender = "Male",   course = "Java" },
        new Student { roll = 2, name = "Priya",  gender = "Female", course = "C#"   },
        new Student { roll = 3, name = "Rahul",  gender = "Male",   course = "Python"}
    };

    return View(students);
}
```

```html
<!-- View — receive the list -->
@model IEnumerable<MyMvcApp.Models.Student>   ← IEnumerable for a list

<h1>Student Details</h1>
<hr />
<div class="container">
@{
    foreach (var item in Model)   ← Loop through each student
    {
        <h3>Roll No: @item.roll</h3>
        <h3>Name:    @item.name</h3>
        <h3>Gender:  @item.gender</h3>
        <h3>Course:  @item.course</h3>
        <hr />
    }
}
</div>
```

### ✅ Why Model is the Best Way

```
With Model (Strongly Typed)          With ViewData/ViewBag (Loosely Typed)
────────────────────────────         ─────────────────────────────────────
@Model.prodName                      @ViewData["prodNaem"]   ← typo!
                                 
↑ Compiler shows error immediately   ↑ No error — just shows empty at runtime
  if you misspell "prodName"           Very hard to debug
  
✅ IntelliSense/autocomplete works   ❌ No autocomplete
✅ Compile-time checking             ❌ Runtime errors only
✅ Clean, readable code              ❌ Need to cast types manually
```

## 4. Method 2 — ViewData

> **ViewData** is a **dictionary** (like a bag of labelled slots).
>
> You store data using a string key in the Controller, and read it with the same key in the View.
>
> It is **loosely typed** — you must cast values to the right type when reading.

### How a Dictionary Works (Simple Analogy)

```
ViewData is like a locker system:

  Controller assigns lockers:           View opens lockers:
  ┌─────────────────────────┐           ┌─────────────────────────┐
  │  ["prodId"]   =  1      │           │  @ViewData["prodId"]    │
  │  ["prodName"] = "i5"    │ ────────► │  @ViewData["prodName"]  │
  │  ["prodPrice"]= 15000   │           │  @ViewData["prodPrice"] │
  │  ["prodQty"]  = 20      │           │  @ViewData["prodQty"]   │
  └─────────────────────────┘           └─────────────────────────┘
        KEY  =  VALUE                        Same KEY to get VALUE
```

### Step 1 — Controller sets ViewData

```csharp
// Controllers/ProductController.cs
public IActionResult Index()
{
    ViewData["prodId"]    = 1;
    ViewData["prodName"]  = "Intel i5 CPU";
    ViewData["prodPrice"] = 15000;
    ViewData["prodQty"]   = 20;

    return View();   // ← No object passed — data is in ViewData
}
```

### Step 2 — View reads ViewData

```html
<!-- Views/Product/Index.cshtml -->
@{
    ViewData["Title"] = "Home Page";
}

@{
    // Must CAST because ViewData stores everything as 'object'
    int price = Convert.ToInt32(ViewData["prodPrice"]);
    int qty   = Convert.ToInt32(ViewData["prodQty"]);
    int total = price * qty;   // You can do calculations in the View
}

<div class="container">
    <h3>Product Id:    @ViewData["prodId"]   </h3>
    <h3>Product Name:  @ViewData["prodName"] </h3>
    <h3>Product Price: @ViewData["prodPrice"]</h3>
    <h3>Product Qty:   @ViewData["prodQty"]  </h3>
    <h3>Stock Value:   @total                </h3>
</div>
```

### ⚠️ The Casting Problem

```csharp
// ViewData stores EVERYTHING as type 'object'

ViewData["prodPrice"] = 15000;   // stored as object, not int

// In View — if you try to do math without casting:
int total = ViewData["prodPrice"] * ViewData["prodQty"];  // ❌ ERROR
// Can't multiply two 'object' types

// You MUST cast first:
int price = Convert.ToInt32(ViewData["prodPrice"]);       // ✅ Correct
int qty   = Convert.ToInt32(ViewData["prodQty"]);
int total = price * qty;                                  // ✅ Works
```

### Key Facts about ViewData

```
✔ Stores data as key-value pairs (like a Dictionary<string, object>)
✔ Key must be a string:  ViewData["myKey"]
✔ Can store any value: string, int, list, object
✔ Only lives for ONE request — gone after View renders
✗ Requires type casting when reading
✗ No compile-time checking — typos cause silent null values
✗ No IntelliSense support for keys
```

## 5. Method 3 — ViewBag

> **ViewBag** is the same as ViewData internally, but accessed using **dot notation** instead of dictionary keys.
>
> It is `dynamic` — you make up property names on the fly without defining them anywhere first.
>
> No casting needed when reading!

### ViewData vs ViewBag — Same Thing, Different Syntax

```
ViewData (dictionary style)         ViewBag (dot notation style)
──────────────────────────          ────────────────────────────
ViewData["Name"] = "Akash"          ViewBag.Name = "Akash"
ViewData["Fee"]  = 6500             ViewBag.Fee  = 6500

@ViewData["Name"]                   @ViewBag.Name
@ViewData["Fee"]                    @ViewBag.Fee

They share the same underlying data!
ViewData["Name"] == ViewBag.Name  → true
```

### Step 1 — Controller sets ViewBag

```csharp
// Controllers/StudentController.cs
public IActionResult Index()
{
    ViewBag.Roll   = 1;
    ViewBag.Name   = "Akash Kumar";
    ViewBag.Course = "Java";
    ViewBag.Fee    = 6500;

    return View();   // No object passed
}
```

### Step 2 — View reads ViewBag

```html
<!-- Views/Student/Index.cshtml -->
<div class="container">
    <h2>Roll:   @ViewBag.Roll  </h2>
    <h2>Name:   @ViewBag.Name  </h2>
    <h2>Course: @ViewBag.Course</h2>
    <h2>Fee:    @ViewBag.Fee   </h2>
</div>
```

### ViewBag in Action — No Casting Needed!

```html
@{
    // ViewBag.Num1 and ViewBag.Num2 were set in the Controller
    // No casting needed — ViewBag handles it dynamically
    int res = ViewBag.Num1 + ViewBag.Num2;
}

<div class="container">
    <h2>Roll:   @ViewBag.Roll  </h2>
    <h2>Name:   @ViewBag.Name  </h2>
    <h2>Result: @res           </h2>
</div>
```

### How ViewBag Works Internally

```
ViewBag is a dynamic wrapper over ViewData.
They share the SAME storage — just different ways to access it.

Controller:
  ViewBag.Name = "Akash"
       ↕  same data
  ViewData["Name"] = "Akash"

View:
  @ViewBag.Name       → "Akash"  ✅
  @ViewData["Name"]   → "Akash"  ✅   (both work!)
```

### Key Facts about ViewBag

```
✔ Uses dot notation — cleaner, more readable syntax
✔ No casting needed (dynamic handles it)
✔ Can create any property name on the fly
✔ Shares storage with ViewData — they are interchangeable
✔ Only lives for ONE request
✗ No compile-time checking
✗ No IntelliSense (dynamic means no autocomplete)
✗ If you misspell the property name, you get null silently
```

## 6. Method 4 — TempData

> **TempData** is special. 
>
> Unlike Model, ViewData, and ViewBag which  **die after one request** , TempData  **survives one redirect** .
>
> It is stored in **session state** behind the scenes.
>
> After you read TempData in the next request, it is  **automatically deleted** .

### The Problem TempData Solves

```
SCENARIO: User submits a form → Controller saves data → Redirects to Index page
          We want to show "Saved successfully!" on the Index page

Without TempData:
  POST /Student/Create
       │
       ├── ViewBag.Message = "Saved!"   ← Set here
       │
       └── return RedirectToAction("Index");  ← NEW request starts
                    │
                    ▼
            GET /Student/Index
                    │
                    └── @ViewBag.Message   → ❌ EMPTY — ViewBag died on redirect!

With TempData:
  POST /Student/Create
       │
       ├── TempData["Message"] = "Saved!"  ← Stored in session
       │
       └── return RedirectToAction("Index");  ← Redirect
                    │
                    ▼
            GET /Student/Index
                    │
                    └── @TempData["Message"]  → ✅ "Saved!" — survived!
                                                  (then auto-deleted)
```

### Lifetime Visualised

```
Request 1 (POST)            Request 2 (GET after redirect)    Request 3
──────────────────          ──────────────────────────────    ──────────
TempData["msg"]             TempData["msg"]                   TempData["msg"]
   = "Saved!"      ──────►     = "Saved!" ✅ readable  ──►      = null ❌ deleted
                                           after read,
                                           auto-cleaned

ViewBag.msg                 ViewBag.msg
   = "Saved!"      ──────►     = null ❌ already gone
```

### Step 1 — Controller sets TempData before redirect

```csharp
// Controllers/StudentController.cs

[HttpPost]
[ValidateAntiForgeryToken]
public IActionResult Create(Student student)
{
    if (ModelState.IsValid)
    {
        _studentBAL.AddStudent(student);

        // Set TempData BEFORE redirect
        TempData["Message"] = "Student added successfully!";
        TempData["Type"]    = "success";   // Can store multiple keys

        return RedirectToAction("Index");   // ← Redirect to another action
    }

    return View(student);
}

[HttpPost]
[ValidateAntiForgeryToken]
public IActionResult Delete(int id)
{
    _studentBAL.DeleteStudent(id);

    TempData["Message"] = "Student deleted!";
    TempData["Type"]    = "danger";

    return RedirectToAction("Index");
}
```

### Step 2 — View reads TempData

```html
<!-- Views/Student/Index.cshtml -->

@* Check if TempData message exists before showing *@
@if (TempData["Message"] != null)
{
    <div class="alert alert-@TempData["Type"]">
        @TempData["Message"]
    </div>
}

<h2>Student List</h2>
<!-- rest of the page ... -->
```

### TempData Keep() — Read Without Deleting

By default, TempData is deleted as soon as you read it. If you need it to survive another request (e.g., you read it but want to use it again), use `.Keep()`:

```csharp
// In View or Controller:
TempData.Keep("Message");   // Keep just this key
TempData.Keep();            // Keep ALL keys
```

### TempData Peek() — Read Without Marking for Deletion

```csharp
// Read the value but DON'T mark it for deletion
var msg = TempData.Peek("Message") as string;
// Message still available for next read
```

### Key Facts about TempData

```
✔ Survives ONE redirect (stored in session)
✔ Perfect for flash messages after form submit / redirect
✔ Automatically deleted after being read
✔ Can store multiple keys like a dictionary
✔ Use .Keep() to preserve across more than one request
✗ Loosely typed — need to cast to string, int, etc.
✗ Requires session to be enabled (AddSession in Program.cs)
✗ Only meant for short-lived messages, not complex data
```

## 7. Side-by-Side Comparison

### Full Example — Same Data, All 4 Methods

**Controller:**

```csharp
public IActionResult CompareAll()
{
    // ── Method 1: Model ──
    var product = new Product { prodId = 1, prodName = "Intel i5", prodPrice = 15000 };

    // ── Method 2: ViewData ──
    ViewData["prodId"]    = 1;
    ViewData["prodName"]  = "Intel i5";
    ViewData["prodPrice"] = 15000;

    // ── Method 3: ViewBag ──
    ViewBag.prodId    = 1;
    ViewBag.prodName  = "Intel i5";
    ViewBag.prodPrice = 15000;

    // ── Method 4: TempData (survives redirect only) ──
    TempData["Message"] = "Data loaded!";

    return View(product);  // Model is passed as argument
}
```

**View:**

```html
@model MyMvcApp.Models.Product

<!-- Method 1: Model — clean, typed, IntelliSense works -->
<h3>@Model.prodId</h3>
<h3>@Model.prodName</h3>
<h3>@Model.prodPrice</h3>

<!-- Method 2: ViewData — need to use string key -->
<h3>@ViewData["prodId"]</h3>
<h3>@ViewData["prodName"]</h3>
<h3>@ViewData["prodPrice"]</h3>

<!-- Method 3: ViewBag — dot notation, no casting -->
<h3>@ViewBag.prodId</h3>
<h3>@ViewBag.prodName</h3>
<h3>@ViewBag.prodPrice</h3>

<!-- Method 4: TempData — for messages across redirects -->
@if (TempData["Message"] != null)
{
    <div class="alert alert-info">@TempData["Message"]</div>
}
```

## 8. When to Use Which?

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       DECISION GUIDE                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Do you need to pass the MAIN data to display on the page?             │
│   (e.g., a product, a student, a list of records)                       │
│                    │                                                     │
│                    └── ✅ Use MODEL                                     │
│                                                                         │
│   Do you need to pass EXTRA small data alongside the main model?        │
│   (e.g., page title, a count, a dropdown list)                          │
│                    │                                                     │
│                    ├── ViewData["Title"] = "Products";                  │
│                    └── ViewBag.Title = "Products";   (either is fine)   │
│                                                                         │
│   Do you need to show a message AFTER a redirect?                       │
│   (e.g., "Record saved!" after POST → Redirect → GET)                   │
│                    │                                                     │
│                    └── ✅ Use TEMPDATA                                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### Real-World Usage Pattern

```csharp
// Most real controllers look like this — a MIX of methods

public IActionResult Index()
{
    // Model — main data for the page
    var students = _bal.GetAllStudents();

    // ViewBag/ViewData — small extra bits
    ViewBag.PageTitle    = "Student List";
    ViewData["Count"]    = students.Count;

    return View(students);  // Model passed here
}

[HttpPost]
public IActionResult Create(Student s)
{
    _bal.AddStudent(s);

    // TempData — survives the redirect below
    TempData["Message"] = "Student added!";

    return RedirectToAction("Index");
}
```

```html
<!-- Index.cshtml -->
@model IEnumerable<Student>

<h1>@ViewBag.PageTitle</h1>          ← from ViewBag
<p>Total: @ViewData["Count"]</p>    ← from ViewData

@if (TempData["Message"] != null)   ← from TempData (after redirect)
{
    <div class="alert alert-success">@TempData["Message"]</div>
}

@foreach (var s in Model)           ← from Model
{
    <p>@s.name</p>
}
```

## 9. Quick Cheat Sheet

```
═══════════════════════════════════════════════════════════════
  MODEL
═══════════════════════════════════════════════════════════════
  Controller:   return View(myObject);
                return View(myList);

  View top:     @model MyApp.Models.Product
                @model IEnumerable<MyApp.Models.Student>

  View use:     @Model.prodName
                @foreach (var item in Model) { @item.name }

  Best for:     Main page data. Always prefer this.

═══════════════════════════════════════════════════════════════
  VIEWDATA
═══════════════════════════════════════════════════════════════
  Controller:   ViewData["Title"] = "My Page";
                ViewData["Count"] = 42;

  View use:     @ViewData["Title"]
                @Convert.ToInt32(ViewData["Count"])  ← cast needed

  Best for:     Extra small data alongside a Model.

═══════════════════════════════════════════════════════════════
  VIEWBAG
═══════════════════════════════════════════════════════════════
  Controller:   ViewBag.Title = "My Page";
                ViewBag.Count = 42;

  View use:     @ViewBag.Title
                @ViewBag.Count   ← no casting needed

  Best for:     Same as ViewData. Cleaner syntax. Pick one style.

═══════════════════════════════════════════════════════════════
  TEMPDATA
═══════════════════════════════════════════════════════════════
  Controller:   TempData["Message"] = "Saved!";
                return RedirectToAction("Index");

  View use:     @if (TempData["Message"] != null) {
                    <div>@TempData["Message"]</div>
                }

  Special:      TempData.Keep("Message");  ← survive one more request
                TempData.Peek("Message");  ← read without deleting

  Best for:     Flash messages after form submit + redirect.

═══════════════════════════════════════════════════════════════
  REMEMBER
═══════════════════════════════════════════════════════════════
  ViewData and ViewBag share the same data store.
  ViewData["X"] == ViewBag.X  →  true

  Model    → Compile-time safe, IntelliSense, best practice
  ViewData → String key access, casting required
  ViewBag  → Dot notation, no casting, dynamic
```
