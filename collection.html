Collections Overview

C# collections live in two main namespaces: System.Collections.Generic (type-safe, modern) and System.Collections (legacy, non-generic). Always prefer generic collections.

C# Collection Hierarchy
IEnumerable<T> └── ICollection<T> ├── IList<T> │ ├── List<T> → Dynamic array │ └── T[] → Fixed array ├── ISet<T> │ ├── HashSet<T> → Hash table │ └── SortedSet<T> → Red-Black Tree └── IDictionary<K,V> ├── Dictionary<K,V> → Hash table └── SortedDictionary → Red-Black Tree Other important ├── Stack<T> → LIFO (array-backed) ├── Queue<T> → FIFO (circular array) ├── LinkedList<T> → Doubly linked list ├── PriorityQueue<T,P> → Min-Heap (.NET 6+) └── Span<T> / Memory<T>→ Stack/heap slices
Key Insight Every generic collection implements IEnumerable<T>, meaning you can use foreach and all LINQ operators on all of them.

Arrays — T[]

The most primitive, most performant collection. Fixed size, contiguous memory block. Zero-indexed. A 2D array is laid out in row-major order in memory.

O(1) random access — CPU can jump directly to any element 

// Declaration & Initialization
int[] arr = new int[5];
int[] arr2 = { 10, 20, 30, 40, 50 };
var arr3 = new int[] { 1, 2, 3 };

// Multi-dimensional
int[,] matrix = new int[3, 3];  // true 2D — single allocation
int[][] jagged = new int[3][];  // array of arrays — separate allocations

// Sorting & Searching
Array.Sort(arr2);               // IntroSort (QuickSort + HeapSort + InsertionSort)
int idx = Array.BinarySearch(arr2, 30);  // O(log n) — array must be sorted

// Copy
int[] copy = (int[])arr2.Clone();   // shallow copy
Array.Copy(arr2, copy, arr2.Length); // same as above

// Spans over arrays — zero-copy slicing
Span<int> slice = arr2.AsSpan(1, 3);  // {20, 30, 40} — no allocation
Performance Note Arrays have the best cache locality of all collections — the CPU prefetcher loves sequential memory. For tight loops over known-size data, raw arrays beat List<T> by ~10–20%.

List<T> — Dynamic Array

The most-used collection in C#. Internally it's a plain T[] array with a tracked _size counter. When you add beyond capacity it allocates a new array of double the size and copies everything over — this is called growth/resize.

The Growth Algorithm — How Resizing Works

List<T> Capacity Doubling
Count reaches Capacity → new array allocated (capacity × 2)
Add #1-4:
Count=4, Cap=4
Add #5 → RESIZE!
Count=5, Cap=8+new array copy
Add #6-8:
Count=8, Cap=8
Add #9 → RESIZE again!
Count=9, Cap=16
Growth sequence: 0 → 4 → 8 → 16 → 32 → 64 → 128 → ... (doubles each time)
Amortized O(1) Add — each element is copied at most twice on average
var list = new List<int>();         // Capacity=0
var list2 = new List<int>(100);     // Capacity=100 (pre-allocate!)

list.Add(10);                       // O(1) amortized
list.Insert(0, 5);                 // O(n) — shifts all elements right
list.Remove(10);                    // O(n) — linear search + shift
list.RemoveAt(0);                  // O(n) — shift left
int x = list[2];                    // O(1) — direct index
list.Contains(10);                  // O(n) — linear scan
list.Sort();                         // O(n log n) — IntroSort
list.BinarySearch(10);             // O(log n) — must be sorted

// Avoid resizing: set capacity upfront if size is known
list.Capacity = 500;               // single allocation
list.TrimExcess();                  // shrink to Count

// AddRange — efficient bulk add
list.AddRange(new[] { 1,2,3,4,5 }); // single potential resize
Watch Out Insert(0, x) and RemoveAt(0) are O(n) — they shift the entire array. If you frequently insert/remove at the front, use LinkedList<T> or Queue<T> instead.

A doubly-linked list. Each element is a LinkedListNode<T> object with Value, Next, and Previous pointers. No contiguous memory — each node is a separate heap allocation.

var ll = new LinkedList<string>();

LinkedListNode<string> bob = ll.AddLast("Bob");    // O(1)
ll.AddFirst("Alice");                              // O(1)
ll.AddAfter(bob, "Carol");                          // O(1) — given the node

// Traversal — O(n)
LinkedListNode<string> node = ll.First;
while (node != null) {
    Console.WriteLine(node.Value);
    node = node.Next;
}

ll.Remove(bob);  // O(1) — if you hold the node reference
ll.Remove("Alice");  // O(n) — must search first

✓ USE LinkedList when

  • Frequent inserts/removes at both ends
  • Frequent inserts/removes in the middle (given a node ref)
  • Implementing undo/redo stacks
  • LRU cache implementation

✗ AVOID LinkedList when

  • Random access is needed (O(n) each)
  • Memory is tight (each node = heap alloc)
  • Cache performance matters (poor locality)
  • LINQ-heavy code (prefer List)

var dict = new Dictionary<string, int>();
var dict2 = new Dictionary<string, int>(100);  // pre-size capacity

// Basic ops — all O(1) average
dict["Alice"] = 25;         // Add or update
dict.Add("Bob", 19);        // Throws if key exists
bool has = dict.ContainsKey("Alice");   // O(1)

// Safe get — TryGetValue is the preferred pattern
if (dict.TryGetValue("Alice", out int age)) {
    Console.WriteLine($"Alice is {age}");
}

// GetOrAdd pattern (no ConcurrentDictionary needed here)
ref int val = ref CollectionsMarshal.GetValueRefOrAddDefault(dict, "count", out _);
val++;  // increment in-place — no double lookup

// Iterate
foreach (var (key, value) in dict) { }
foreach (KeyValuePair<string,int> kv in dict) { }

// Custom equality comparer (e.g., case-insensitive keys)
var ci = new Dictionary<string, int>(StringComparer.OrdinalIgnoreCase);

// Implementing custom GetHashCode for struct keys:
record struct Point(int X, int Y);  // records auto-generate GetHashCode
Critical: Custom Types as Keys If you use a custom class as a Dictionary key, you MUST override both GetHashCode() and Equals(). If two objects are equal, they MUST return the same hash code. Violating this breaks the dictionary entirely.
Pro Tip: CollectionsMarshal CollectionsMarshal.GetValueRefOrNullRef() (.NET 6+) gives a ref to the value in-place — avoids double-lookup when you need to read-then-update, which is a very common pattern.

HashSet<T>

Essentially a Dictionary with no values — just keys. Uses the same hash table internally. Guarantees unique elements and provides mathematical set operations in O(n).

var set = new HashSet<int> { 1, 2, 3, 4, 5 };
var other = new HashSet<int> { 3, 4, 5, 6, 7 };

set.Add(6);          // O(1) — returns false if already present
set.Contains(3);    // O(1) — fastest "contains" available
set.Remove(2);      // O(1)

// Set operations — all O(n)
set.UnionWith(other);            // {1,2,3,4,5,6,7}
set.IntersectWith(other);        // {3,4,5}
set.ExceptWith(other);           // {1,2}  (set minus other)
set.SymmetricExceptWith(other);  // {1,2,6,7}  (XOR)

bool sub = set.IsSubsetOf(other);
bool sup = set.IsSupersetOf(other);
bool over = set.Overlaps(other);     // any common element?

// Deduplication — fastest way to unique a collection
var dupes = new[] { 1,2,2,3,3,3 };
var unique = new HashSet<int>(dupes);  // {1,2,3}
HashSet vs List.Contains HashSet.Contains() is O(1). List.Contains() is O(n). If you're checking membership frequently, always use a HashSet. For 1M elements: HashSet ≈ 100ns, List ≈ 500ms.

Stack<T> — LIFO

Last In, First Out. Internally backed by a T[] array with a top index pointer. Push/Pop are O(1) amortized. The "top" grows upward in the array.

Stack<string> — LIFO Visualization
Method_A()
Method_B()
Method_C()
Method_D()                      
Call Stack (example)
Push(x): arr[++top] = x
Pop()  : return arr[top--]
Peek() : return arr[top]
Internal array doubles when full
var stack = new Stack<string>();

stack.Push("first");   // O(1)
stack.Push("second");
stack.Push("third");

string top = stack.Peek();       // "third" — peek without remove, O(1)
string popped = stack.Pop();     // "third" — O(1)
bool found = stack.TryPop(out string val);  // safe pop

// Real use: bracket matching, undo/redo, expression evaluation
static bool IsBalanced(string s) {
    var stk = new Stack<char>();
    foreach (char c in s) {
        if (c == '(') stk.Push(c);
        else if (c == ')') {
            if (stk.Count == 0) return false;
            stk.Pop();
        }
    }
    return stk.Count == 0;
}

Queue<T> — FIFO

First In, First Out. Backed by a circular ring buffer (T[] array with head/tail indices that wrap around). This avoids the O(n) shifting problem of a naïve array queue.

Queue<int> — Circular Ring Buffer Internals
10
HEAD (Dequeue)
20
 
30
 
40
 
50
TAIL (Enqueue)
_head and _tail are indices into the internal array
Enqueue: arr[_tail] = item; _tail = (_tail+1) % capacity → O(1)
Dequeue: item = arr[_head]; _head = (_head+1) % capacity → O(1)
When full: allocate new array (2× size), copy from _head around 
var queue = new Queue<int>();

queue.Enqueue(10);   // Add to tail — O(1)
queue.Enqueue(20);
queue.Enqueue(30);

int front = queue.Peek();         // 10 — peek without remove
int removed = queue.Dequeue();   // 10 — remove from front O(1)
bool ok = queue.TryDequeue(out int val);  // safe dequeue

// Real use: BFS graph traversal
void BFS(Graph g, int start) {
    var visited = new HashSet<int>();
    var q = new Queue<int>();
    q.Enqueue(start);
    while (q.Count > 0) {
        int node = q.Dequeue();
        if (!visited.Add(node)) continue;
        foreach (int neighbor in g.Neighbors(node))
            q.Enqueue(neighbor);
    }
}

SortedSet<T> & SortedDictionary

Both are backed by a Red-Black Tree — a self-balancing binary search tree. O(log n) for all operations, but always sorted. No hash function needed.

Red-Black Tree Structure

SortedSet<int> { 1,2,3,4,5,6,7 } → Red-Black Tree
4
↙↘
2
6
↙↘↙↘
1
3

Rules: (1) Every node is red or black  (2) Root is always black
(3) No two adjacent red nodes  (4) All paths to leaves have same black-depth
→ Tree height ≤ 2·log₂(n+1) → guaranteed O(log n) 
var ss = new SortedSet<int> { 5, 2, 8, 1, 9 };

// Always sorted: {1, 2, 5, 8, 9}
ss.Add(3);     // O(log n)
ss.Remove(5);  // O(log n)
ss.Contains(8); // O(log n) ← slower than HashSet O(1)!

// Unique power: range queries — impossible with HashSet
SortedSet<int> range = ss.GetViewBetween(2, 8); // O(log n)
int min = ss.Min;   // O(1) cached
int max = ss.Max;   // O(1) cached

// SortedDictionary — same tree, just stores K/V pairs
var sd = new SortedDictionary<string, int>();
sd["Charlie"] = 1;
sd["Alice"] = 2;
sd["Bob"] = 3;
// Iteration yields: Alice, Bob, Charlie (alphabetical)

// SortedList — like SortedDictionary but array-backed (less memory,
//              faster iteration, O(n) insert/remove)
var sl = new SortedList<string, int>();

HashSet / Dictionary

  • O(1) average lookup
  • No ordering guarantee
  • More memory (hash table)
  • Best for: frequency, membership

SortedSet / SortedDictionary

  • O(log n) all ops
  • Always sorted
  • Range queries supported
  • Best for: leaderboards, sliding window

PriorityQueue<TElement, TPriority>

Added in .NET 6. Backed by a 4-ary min-heap (quaternary heap, not binary). Always Dequeues the element with the lowest priority value. O(log n) enqueue/dequeue.

Binary Min-Heap (simplified view) — smallest always at root
1
↙↘
3
4
↙↘↙↘
5
8
6
9
Stored in array: [1, 3, 4, 5, 8, 6, 9] (BFS order)
Parent of i = (i-1)/2    Children of i = 2i+1, 2i+2
Enqueue: add to end, bubble UP → O(log n)
Dequeue: remove root, move last to root, sift DOWN → O(log n) 
var pq = new PriorityQueue<string, int>(); // (element, priority)

pq.Enqueue("Task C", 3);  // priority 3
pq.Enqueue("Task A", 1);  // priority 1 — lowest = highest urgency
pq.Enqueue("Task B", 2);  // priority 2

string next = pq.Dequeue();  // "Task A" (priority 1)
next = pq.Dequeue();         // "Task B" (priority 2)

pq.TryPeek(out string el, out int pri); // peek without remove

// Dijkstra's shortest path — classic PQ use case
var dist = new Dictionary<int, int>();
var pqDijk = new PriorityQueue<int, int>();  // (node, distance)
pqDijk.Enqueue(0, 0); // start node, distance 0

Concurrent Collections

In System.Collections.Concurrent. Thread-safe without external locking. Use in multi-threaded scenarios like producer-consumer patterns, parallel workloads, etc.

ConcurrentDictionary<K,V>

  • Fine-grained locking (stripe locks)
  • GetOrAdd, AddOrUpdate atomic
  • Better than lock+Dictionary

ConcurrentQueue<T>

  • Lock-free segments algorithm
  • TryDequeue instead of Dequeue
  • Perfect for work queues

ConcurrentBag<T>

  • Unordered, per-thread local lists
  • Fast when same thread add+remove
  • Thread-local storage trick

BlockingCollection<T>

  • Wraps any IProducerConsumerCollection
  • Blocks Add when full, Take when empty
  • BoundedCapacity throttling
// ConcurrentDictionary — atomic patterns
var cd = new ConcurrentDictionary<string, int>();

// Atomic increment — no race condition
cd.AddOrUpdate("visits", 1, (key, old) => old + 1);

// Lazy init — only computes if key missing
var val = cd.GetOrAdd("cache_key", k => ExpensiveCompute(k));

// ConcurrentQueue — producer/consumer
var q = new ConcurrentQueue<string>();
Task.Run(() => q.Enqueue("work item"));  // producer
Task.Run(() => {                              // consumer
    if (q.TryDequeue(out string item))
        Process(item);
});

// BlockingCollection — bounded producer/consumer pipeline
using var bc = new BlockingCollection<int>(boundedCapacity: 100);
Task.Run(() => {
    for (int i = 0; i < 1000; i++) bc.Add(i);  // blocks if full
    bc.CompleteAdding();
});
foreach (int item in bc.GetConsumingEnumerable())
    Process(item);  // blocks until item available
Watch Out: GetOrAdd Race GetOrAdd on ConcurrentDictionary is NOT atomic — the factory may be called multiple times. For truly atomic lazy init, use Lazy<T> as the value type.

Immutable Collections

In System.Collections.Immutable NuGet. Every "mutation" creates a new version sharing structure with the old one via persistent data structures (structural sharing). Safe for functional patterns and multi-threading without locks.

Shared nodes (dimmed) are reused — only new path is allocated
ImmutableList internally uses a balanced binary tree, not an array → O(log n) ops 
using System.Collections.Immutable;

ImmutableList<int> v1 = ImmutableList<int>.Empty;
ImmutableList<int> v2 = v1.Add(1).Add(2).Add(3);  // v1 unchanged!
ImmutableList<int> v3 = v2.Remove(2);  // v2 still {1,2,3}

// Bulk build with Builder — avoids O(n log n) individual adds
var builder = ImmutableList<int>.CreateBuilder();
builder.Add(1); builder.Add(2); builder.Add(3);
ImmutableList<int> final = builder.ToImmutable(); // O(n) build

// ImmutableDictionary — perfect for config / state snapshots
var config = ImmutableDictionary<string,string>.Empty
    .Add("env", "prod")
    .Add("version", "2.1");

// ImmutableArray — struct wrapping T[], zero overhead vs array
ImmutableArray<int> ia = ImmutableArray.Create(1,2,3);

Span<T> & Memory<T>

.NET Core 2.1+ Not a collection per se, but the most important performance tool when working with them. A ref struct that represents a contiguous region of memory — over arrays, stack, or unmanaged memory — with zero copying.

// Span over array — no allocation, just a pointer+length view
int[] arr = { 1, 2, 3, 4, 5 };
Span<int> span = arr.AsSpan();
Span<int> slice = span[1..4];  // {2,3,4} — no allocation!
slice[0] = 99;               // modifies original array!

// Stackalloc — allocate on stack (zero GC pressure)
Span<int> stackSpan = stackalloc int[128];  // fast! no heap alloc

// String parsing WITHOUT allocation (huge win for parsers)
string date = "2024-01-15";
ReadOnlySpan<char> dateSpan = date.AsSpan();
ReadOnlySpan<char> year = dateSpan[..4];   // "2024" — no string alloc
int y = int.Parse(year);

// Memory<T> — heap-based, can be stored in fields (unlike Span)
Memory<byte> mem = new byte[1024];
await stream.ReadAsync(mem);  // async-compatible
Span<byte> s = mem.Span;       // get Span when sync needed
Zero-Alloc Parsing Pattern Using ReadOnlySpan<char> for string splitting avoids allocating substring objects. For hot paths processing millions of strings, this can reduce GC pauses by 80%+.

Quick Decision Guide

Need fast lookup by key?

  • → Dictionary<K,V>
  • → HashSet if just membership

Need sorted data + ranges?

  • → SortedSet / SortedDictionary
  • → SortedList if read-heavy

LIFO? (undo, call stack)

  • → Stack<T>

FIFO? (BFS, task queue)

  • → Queue<T>

Priority scheduling?

  • → PriorityQueue<T,P>

Multi-threaded access?

  • → ConcurrentDictionary
  • → ConcurrentQueue / Bag
  • → BlockingCollection

Immutable / functional style?

  • → ImmutableList / Dictionary
  • → ImmutableArray (perf)

Hot path, zero alloc?

  • → Span<T> / stackalloc
  • → ArrayPool<T>.Shared
